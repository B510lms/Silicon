## Q1：你这个项目主要做了什么？

**答：**

## Q2：token 是怎么做登录鉴权的？

**答：**

登录流程是这样的：

1. 用户在登录页输入账号密码，调用 `reqLogin` 接口
2. 后端验证成功后返回 token，我把 token 存到两个地方：
   - Pinia 的 `userStore.token`（内存中，响应式）
   - `localStorage`（持久化，通过 `SET_TOKEN` 工具方法）
3. 之后每次请求，在 axios 请求拦截器里从 `userStore` 取出 token，设置到请求头的 `token` 字段
4. 路由守卫 `router.beforeEach` 里判断：有 token 放行，没 token 跳登录页

**对应文件：**
- `src/views/login/index.vue` - 登录页面，表单校验 + 调用 `userLogin`
- `src/store/modules/user.ts` - `userLogin` action：存 token 到 Pinia + localStorage
- `src/utils/request.ts` - 请求拦截器携带 token：`config.headers.token = userStore.token`
- `src/permisstion.ts` - 路由守卫判断 token

## Q3：token 过期怎么处理？

**答：**

分两个层面处理：

**1. 请求层面**：在 axios 响应拦截器里判断后端返回的 code
- 后端约定 `code === 206` 表示"无效的 Token"
- 这时候会 `ElMessage.error` 提示，然后 `Promise.reject`

**2. 路由层面**：在 `permisstion.ts` 的路由守卫中
- 用户已登录（有 token）但访问页面时调用 `userStore.userInfo()` 获取用户信息
- 如果 token 过期，获取用户信息会失败（catch），这时调用 `userLogout()` 清空数据，然后跳转到登录页，并带上 `redirect` 参数方便登录后回跳

```ts
// permisstion.ts 核心逻辑
try {
  await userStore.userInfo()
  next({ ...to }) // 重新导航解决异步路由未加载完的问题
} catch (error) {
  await userStore.userLogout()
  next({ path: '/login', query: { redirect: to.path } })
}
```

**对应文件：**
- `src/utils/request.ts` - 响应拦截器：code 非 200 统一报错
- `src/permisstion.ts` - 路由守卫 catch 中处理 token 过期
- `src/store/modules/user.ts` - `userInfo()` 和 `userLogout()`

## Q4：你 axios 是怎么封装的？

**答：**

我封装了 `src/utils/request.ts`，主要做了这几件事：

**1. 创建实例**
- `baseURL` 从环境变量 `import.meta.env.VITE_APP_BASE_API` 读取（开发时是 `/api`）
- `timeout` 设为 5000ms

**2. 请求拦截器**
- 从 Pinia 的 `userStore` 获取 token
- 有 token 就设置到请求头：`config.headers.token = userStore.token`

**3. 响应拦截器**
- **成功回调**：判断后端返回的 `code`，只有 `code === 200` 才返回数据，否则 `ElMessage.error` 并 `Promise.reject`
- **失败回调**（网络错误）：根据 HTTP 状态码统一提示：401(token过期)、403(无权访问)、404(地址错误)、500(服务器问题)

后端约定的状态码：
- 200: 成功
- 201: 参数错误
- 202: 用户名已存在
- 203: 用户名不存在
- 204: 用户名或密码错误
- 205: 服务繁忙
- 206: 无效的 Token
- 207: 无权访问
- 208: 该节点下有子节点不可删除

**对应文件：**
- `src/utils/request.ts` - 完整封装
- `.env.development` / `.env.production` - 环境变量配置 `VITE_APP_BASE_API = '/api'`

## Q5：动态路由是怎么实现的？

**答：**

动态路由的核心思路是：**后端返回用户有权访问的路由 name 列表，前端据此过滤出对应路由组件，用 `router.addRoute()` 动态注册**。

具体流程：

1. **定义路由**：在 `routes.ts` 中定义好三类路由
   - `constantRoute`：常量路由（登录、首页、404、数据大屏），所有人都能访问
   - `asnycRoute`：异步路由（权限管理、商品管理），需要权限控制
   - `anyRoute`：任意路由，匹配不到的都定向到 404

2. **登录后获取权限**：`userStore.userInfo()` 调用接口获取用户信息，后端返回 `routes` 数组（如 `['Acl', 'User', 'Product', 'Trademark']`）

3. **过滤路由**：用 `filterAsyncRoute` 函数递归过滤异步路由，只保留 `routes` 数组中包含的 name

4. **动态注册**：通过 `router.addRoute(route)` 把过滤后的路由注册到路由器

5. **生成菜单**：过滤后的路由数组存到 `menuRoutes`，用于渲染侧边栏菜单

**对应文件：**
- `src/router/routes.ts` - 三类路由定义
- `src/store/modules/user.ts` - `userInfo()` action：获取权限 + 过滤路由 + `addRoute()`
- `src/store/modules/user.ts` - `filterAsyncRoute` 函数：递归过滤

## Q6：路由和菜单怎么关联？

**答：**

路由和菜单是**同源**的，用同一套路由数据驱动。

1. 路由的 `meta` 信息里定义了菜单需要的字段：`title`（菜单标题）、`icon`（Element Plus 图标名）、`hidden`（是否隐藏）

2. 登录后，`userStore.menuRoutes` 存储了当前用户可见的全部路由（常量路由 + 过滤后的异步路由 + 任意路由）

3. `layout/index.vue` 里的侧边栏用 `el-menu` 组件，通过 `:default-active="route.path"` 绑定当前路由

4. 菜单渲染交给 `Menu` 递归组件，它接收 `menuList`（即 `menuRoutes`），根据路由结构递归渲染：
   - 无子路由 → `el-menu-item`
   - 只有一个子路由 → 直接渲染该子路由（不显示父级）
   - 多个子路由 → `el-sub-menu`，内部递归调用 `<Menu>`

5. 图标通过 `<component :is="item.meta.icon">` 动态渲染 Element Plus 图标组件

**对应文件：**
- `src/router/routes.ts` - 路由 meta 信息定义
- `src/layout/index.vue` - 布局：el-menu 绑定 menuRoutes
- `src/layout/menu/index.vue` - 递归菜单组件
- `src/store/modules/user.ts` - menuRoutes 状态

## Q7：页面刷新后动态路由丢失怎么办？

**答：**

这是动态路由的经典问题。因为 `router.addRoute()` 是运行时添加的，刷新页面后 Vue Router 实例会重置，动态添加的路由就丢了。

**解决方案在 `permisstion.ts` 的路由守卫中：**

1. 用户已登录且有 token，访问任意页面时进入 `beforeEach`
2. 判断是否有 `username`（用户信息）
   - 有：说明已经获取过用户信息，直接放行 `next()`
   - 没有：调用 `userStore.userInfo()` 重新获取用户信息
3. `userInfo()` 内部会重新过滤异步路由、重新 `router.addRoute()`、重新设置 `menuRoutes`
4. 获取成功后用 `next({ ...to })` 重新导航一次（**关键点**：因为异步路由刚 add 完，直接 `next()` 可能路由还没注册好，用 `next({ ...to })` 让路由守卫再执行一次，确保能匹配到新路由）

```ts
// permisstion.ts 核心逻辑
if (username) {
  next() // 已有用户信息，放行
} else {
  await userStore.userInfo() // 重新获取 + addRoute
  next({ ...to }) // 重新导航，解决异步路由未加载完的问题
}
```

**对应文件：**
- `src/permisstion.ts` - 路由守卫核心逻辑
- `src/store/modules/user.ts` - `userInfo()` action

## Q8：除了路由权限，还有做按钮权限吗？

**答：**

做了，用**自定义指令 `v-has`** 实现按钮级权限控制。

实现原理：
1. 后端返回的用户信息中包含 `buttons` 数组（如 `['btn.Trademark.add', 'btn.Trademark.delete']`）
2. 在 `src/directive/has.ts` 中定义全局自定义指令 `v-has`
3. 指令的 `mounted` 钩子中，检查绑定的值是否在 `userStore.buttons` 数组中
4. **如果不包含，直接从 DOM 树移除该元素**：`el.parentNode.removeChild(el)`

使用方式：在模板中写 `v-has="'btn.Trademark.add'"`

```ts
// has.ts
app.directive('has', {
  mounted(el, options) {
    if (!userStore.buttons.includes(options.value)) {
      el.parentNode.removeChild(el)
    }
  }
})
```

**对应文件：**
- `src/directive/has.ts` - 自定义指令实现
- `src/main.ts` - 注册自定义指令 `isHasButton(app)`
- `src/views/product/trademark/index.vue` - 实际使用：`v-has="'btn.Trademark.add'"`
- `src/store/modules/user.ts` - buttons 数组

## Q9：SPU 和 SKU 你怎么理解？

**答：**

这是电商系统的核心概念，我在商品管理模块中完整实现了这两个模块：

- **SPU**（Standard Product Unit，标准产品单位）：表示一个商品的整体概念，是商品信息聚合的最小单位。比如 "iPhone 14" 就是一个 SPU，它包含了商品的通用信息（品牌、名称、描述、图片等），不涉及具体规格。

- **SKU**（Stock Keeping Unit，库存单位）：表示具体可售卖的商品规格，是最小库存单元。比如 "iPhone 14 黑色 128G" 就是一个 SKU，它有具体的颜色、内存等属性，有独立的价格和库存。

**关系**：一个 SPU 下可以有多个 SKU，通过不同的属性组合（颜色、内存、尺寸等）来区分。

在项目中的体现：
- SPU 管理：维护商品的基本信息和规格参数模板
- SKU 管理：基于 SPU 的规格参数生成具体的 SKU，每个 SKU 有独立的库存、价格

**对应文件：**
- `src/views/product/spu/index.vue` - SPU 管理
- `src/views/product/sku/index.vue` - SKU 管理
- `src/router/routes.ts` - 路由定义

## Q10：商品列表分页怎么做？

**答：**

以品牌管理模块为例：

1. 前端维护两个响应式变量：`pageNo`（当前页码）和 `limit`（每页条数），初始值分别是 1 和 3
2. 调用接口时把这两个参数传给后端：`reqHasTrademark(pageNo, limit)`
3. 后端返回 `total`（总条数）和 `records`（当前页数据）
4. 用 Element Plus 的 `el-pagination` 组件绑定：
   - `v-model:current-page="pageNo"` - 当前页
   - `v-model:page-size="limit"` - 每页条数
   - `@size-change="sizeChange"` - 每页条数变化时重新请求
   - `@current-change="getHasTrademark"` - 页码变化时重新请求
   - `:page-sizes="[3, 5, 7, 9]"` - 可切换的每页条数选项
5. 分页器布局：`layout="prev, pager, next, jumper, ->, sizes, total"`

**对应文件：**
- `src/views/product/trademark/index.vue` - 分页器使用和接口调用

## Q13：数据大屏是怎么做的？

**答：**

数据大屏主要解决了两个核心问题：**数据可视化** 和 **屏幕适配**。

**1. 数据可视化**：使用 ECharts 实现各种图表（柱状图、折线图、饼图、地图等），各图表封装成独立子组件（Top、Tourist、Sex、Age、Map、Line、Rank、Year、Counter）

**2. 屏幕适配（重点）**：
   - 设计稿尺寸是 1920 x 1080
   - 大屏容器用 `position: fixed` + `left: 50%; top: 50%` 居中
   - 通过 `transform: scale() translate(-50%, -50%)` 实现等比例缩放
   - `getScale()` 函数计算屏幕宽高比和设计稿宽高比的较小值，取较小值保证内容完整显示
   - 监听 `window.onresize` 实时调整缩放比例

```ts
function getScale(w = 1920, h = 1080) {
  const ww = window.innerWidth / w;
  const wh = window.innerHeight / h;
  return ww < wh ? ww : wh;
}
```

**对应文件：**
- `src/views/screen/index.vue` - 大屏容器和缩放适配
- `src/views/screen/components/` - 各图表子组件

## Q14：你觉得这个项目最难的点是什么？

**答：**

我觉得最难的是**动态路由权限控制**这一块，因为它不是单一功能，而是涉及多个环节的联动：

1. **权限数据处理**：后端返回的是路由 name 列表，前端需要根据这个列表从预定义的全部异步路由中过滤出用户能访问的。这里用了递归过滤函数 `filterAsyncRoute`，要考虑多级嵌套路由的情况。

2. **路由动态注册**：过滤后用 `router.addRoute()` 动态添加，时机要把握好——必须在路由守卫中获取到用户信息后才能添加。

3. **刷新丢失问题**：动态路由刷新会丢，要在 `beforeEach` 中检测用户信息是否存在，不存在就重新获取并 addRoute，然后用 `next({ ...to })` 重新导航。

4. **权限一致性**：路由权限和菜单展示必须保持一致，菜单数据直接来自过滤后的路由数组。

这几个环节环环相扣，一个地方出问题就会导致白屏或者菜单显示不对。

**对应文件：**
- `src/store/modules/user.ts` - filterAsyncRoute + addRoute
- `src/permisstion.ts` - 路由守卫解决刷新问题
- `src/layout/menu/index.vue` - 菜单递归渲染

## Q15：你这个项目有什么亮点？

**答：**

1. **基于角色的动态路由权限控制**：不是简单的路由拦截，而是后端返回权限列表，前端动态过滤路由并 `addRoute` 注册，实现了真正的权限隔离。同时用自定义指令 `v-has` 实现了按钮级权限，粒度更细。

2. **axios 双层拦截封装**：请求拦截器统一携带 token，响应拦截器统一按后端约定的 code 处理（200 放行，其他报错），网络错误按 HTTP 状态码分类提示。

5. **Pinia 状态管理**：相比 Vuex，Pinia 更轻量，API 更简洁，支持 Composition API 风格，且有完整的 TS 类型推导。

6. **暗黑模式和主题色切换**：通过修改 CSS 变量 `--el-color-primary` 和切换 HTML 根节点的 `dark` class 实现。

**对应文件：**

## Q16：路由为什么用 hash 模式？

**答：**

项目用的是 `createWebHashHistory`。

hash 模式的优势是**不需要后端配置**。URL 中 `#` 后面的部分不会发送到服务器，刷新页面时始终请求的是同一个 HTML 文件，服务器只需要提供静态资源即可。

如果用 history 模式，刷新页面时浏览器会向服务器请求当前路径（如 `/product/trademark`），需要后端配置 fallback 到 `index.html`，否则会出现 404。

我们项目部署在 nginx 上，hash 模式更省心。

**对应文件：**
- `src/router/index.ts` - `createWebHashHistory()`

## Q17：跨域是怎么解决的？

**答：**

开发环境用 **Vite 的 proxy 代理**解决跨域。

在 `vite.config.ts` 中配置 `server.proxy`，把以 `/api` 开头的请求代理到后端服务器 `http://192.168.206.128:10086`，同时通过 `rewrite` 去掉 `/api` 前缀：

```ts
server: {
  proxy: {
    [env.VITE_APP_BASE_API]: {
      target: env.VITE_SERVE,  // http://192.168.206.128:10086
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api/, '')
    }
  }
}
```

生产环境没有跨域问题，因为前端打包后的静态文件和后端 API 都部署在同一个域名下（通过 nginx 反向代理）。

**对应文件：**
- `vite.config.ts` - proxy 配置
- `.env.development` - `VITE_SERVE = 'http://192.168.206.128:10086'`

## Q18：Pinia 和 Vuex 有什么区别？

**答：**

本项目用的是 Pinia，主要区别：

1. **API 更简洁**：Pinia 没有 mutations，直接通过 action 修改 state，写法更像普通函数
2. **更好的 TS 支持**：Pinia 天生对 TypeScript 友好，不需要额外的类型封装
3. **体积更小**：Pinia 更轻量
4. **支持多 Store**：Pinia 不需要命名空间，每个 Store 都是独立的模块
5. **Composition API 风格**：可以用类似 setup 的函数式写法定义 Store

本项目分了多个 Store 模块：`user.ts`（用户）、`category.ts`（分类）、`layoutSetting.ts`（布局设置），各自独立管理状态。

**对应文件：**
- `src/store/modules/user.ts`
- `package.json` - `"pinia": "^3.0.4"`

## Q19：登录成功后为什么用 `next({ ...to })` 而不是直接 `next()`？

**答：**

这是为了解决**异步路由加载时序问题**。

场景：用户刷新页面，token 还在，但动态路由已经丢失。此时在 `beforeEach` 中：
1. 检测到 `username` 为空，调用 `userStore.userInfo()`
2. `userInfo()` 内部用 `router.addRoute()` 注册异步路由
3. 如果直接 `next()`，路由守卫放行时，新路由可能还没注册完成，Vue Router 匹配不到路由就会显示空白
4. 用 `next({ ...to })` 会**重新触发一次路由守卫**，这时异步路由已经 add 完了，就能正常匹配到路由了

这是 Vue Router 动态路由的一个经典技巧。

**对应文件：**
- `src/permisstion.ts` - `next({ ...to })`

## Q20：上传图片时怎么携带 token？

**答：**

品牌管理模块中有图片上传功能（上传品牌 LOGO），用的是 Element Plus 的 `el-upload` 组件。

`el-upload` 的 HTTP 请求是自己内部发起的，不走 axios 拦截器，所以需要手动设置请求头：

```ts
const headers = { Token: userStore.token }
```

然后绑定到组件的 `:headers` 属性上：

```vue
<el-upload :headers="headers" action="/api/admin/product/fileUpload">
```

同时在上传前做了校验：`before-upload` 钩子中限制文件格式（png/jpg/gif）和大小（< 4M）。

**对应文件：**
- `src/views/product/trademark/index.vue` - el-upload 组件使用
